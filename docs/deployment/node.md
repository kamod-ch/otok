# Node Deployment

Otok's Phase 1 reference runtime is Node.js 20+ with ESM, Hono's Node server, Vite's client manifest, and separate client/server builds.

## Requirements

- Node.js 20 or newer
- pnpm
- A production install that includes runtime dependencies
- A reverse proxy or platform router for TLS in most deployments

## Build

The default app build runs two Vite builds:

```bash
pnpm build:client
pnpm build:server
```

The client build writes hashed assets and `.vite/manifest.json` to `dist/client`. The server build writes `dist/server/server.js`. `readOtokManifest(import.meta.url)` reads the client manifest at runtime so generated HTML links hashed JS and CSS assets.

## Start

```bash
NODE_ENV=production PORT=3000 HOST=0.0.0.0 pnpm start
```

Supported environment variables in the default template:

| Variable | Default | Purpose |
| --- | --- | --- |
| `PORT` | `3000` | TCP port for the Node server |
| `HOST` | Node default | Optional bind host, for example `0.0.0.0` in containers |
| `NODE_ENV` | unset | Set to `production` in deployments |

The template handles `SIGTERM` and `SIGINT` by closing the Node server before exiting.

## Health checks

`createOtokApp({ health })` registers `/api/health`. The default full template returns:

```json
{ "ok": true, "framework": "otok" }
```

Use this endpoint for container, load balancer, or systemd health checks. Add a separate readiness endpoint through `createOtokApp({ configure })` if your app needs database or queue checks.

## Static assets and caching

When `staticDir` is configured, Otok serves assets under `assetsPath` (`/assets` by default). Static asset responses receive:

```http
Cache-Control: public, max-age=31536000, immutable
```

Override this with `assetCacheControl` if your asset filenames are not content-hashed.

HTML responses are generated per request by the app. Set explicit cache headers in Hono middleware when a route is safe to cache.

## Reverse proxies and forwarded headers

Run Otok behind a proxy for TLS, compression, and platform routing. Configure your proxy to forward:

```http
X-Forwarded-For
X-Forwarded-Proto
Host
```

Read and trust forwarded headers only according to your deployment boundary. Otok does not automatically trust arbitrary forwarded headers.

## Docker

See `examples/deployment/node/` for a multi-stage Docker example.

Typical commands:

```bash
docker compose -f examples/deployment/node/compose.yaml up --build
curl http://localhost:3000/api/health
```

## systemd example

```ini
[Unit]
Description=Otok app
After=network.target

[Service]
WorkingDirectory=/srv/otok-app
Environment=NODE_ENV=production
Environment=PORT=3000
ExecStart=/usr/bin/node dist/server/server.js
Restart=always
RestartSec=5
KillSignal=SIGTERM
TimeoutStopSec=20

[Install]
WantedBy=multi-user.target
```

## Logging

The default template logs startup and shutdown events to stdout/stderr. Use your platform's log collector for structured aggregation. Avoid exposing raw error details in production; `exposeErrorDetails` remains opt-in and should usually stay disabled.

## Production smoke test

The repository includes a smoke test for the playground production build:

```bash
pnpm smoke:node
```

It builds the app, starts the built server, checks `/api/health`, fetches an HTML route, verifies a built static asset and cache header, checks a 404, and shuts the server down.

## Common issues

- **Manifest not found**: run the client build before the server starts and keep `dist/client/.vite/manifest.json` next to `dist/server` as generated by the template.
- **CSS missing**: ensure `clientEntry` matches the client build input (`src/client.ts`) and the manifest is readable.
- **Assets 404**: start the app from the project root or set `staticDir` to the correct runtime path.
- **Port already in use**: set `PORT` to an available value.
- **Container not reachable**: bind `HOST=0.0.0.0`.
